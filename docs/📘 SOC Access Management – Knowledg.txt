🔧 Project Overview

The SOC Access Management project provides a streamlined way for the Security Operations Center (SOC) team to manage access across internal tools and platforms using a combination of:

A CLI tool (access_management.py)
A GUI wrapper (soctool.py)
Modular system-specific scripts (check/grant/remove)
Credential-secured operations (pending integration via Cloud Workstation VM)
📁 Repository Location

Location:
/site/socbot/scripts/ (Cloud Workstation VM)

Primary Folders:

SocTool/ → main folder for all GUI and CLI tools
access_management.py → CLI tool (entry point)
soctool.py → GUI wrapper (calls access_management.py)
Systems/UIs/ → location for individual system integration scripts (e.g., aws.py, jira.py, etc.)
Logs/ → auto-created if not present
Utils/ → shared functions (e.g., rt_api.py)
Secrets/ → will hold local credential files for authentication (set via environment variables or local config)
🔑 Authentication & Credentials (Current Status)

Cloud Workstation VM will act as the credential store.

Each system script (AWS, Jira, Grafana, etc.) will reference locally-stored secrets on the Cloud Workstation VM. These may include:

API keys
Service credentials
LDAP bind credentials
DB user/passwords for Oracle/Snowflake/etc.

🧩 Script Behavior
access_management.py (CLI Tool)
Allows interactive or argument-based control of user access
Handles checking, granting, and removing access
Can be run manually or by GUI tool
Uses subprocess to call individual system scripts located in Systems/UIs/
soctool.py (GUI Tool)
Tkinter-based GUI that wraps the CLI for easier usage
Allows for user-friendly selection of check/remove/grant actions
Passes mock flag and target usernames to CLI
Individual System Scripts (e.g., aws.py, jira.py, etc.)
Each script supports: main(user, creds) and foundAccess(result) interfaces
Can be imported or executed via subprocess

Follows standard creds dict pattern:

creds = {
  "operatorUN": "user123",
  "accessChoice": "check/remove/grant",
  "MOCK_MODE": True/False
}
