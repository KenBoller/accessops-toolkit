## 🧭 Project Overview

The **SOC Access Management Toolkit** allows the Security Operations Center (SOC) to manage user access across 20+ internal systems using a unified process:

- ✅ **CLI Tool** (`access_management.py`)
- ✅ **GUI Wrapper** (`soctool.py`)
- ✅ Modular per-system scripts (located in `Systems/UIs/` and `Systems/DBs/`)
- 🔐 Credential integration pending (via **SOC Cloud Workstation**)

## 🛠️ How to Run It

### 🔍 MOCK Mode (Testing Only)
Used to safely simulate system behavior without making real API calls or DB changes.

# Run CLI with mock flag
```bash
python access_management.py --mock
```

# Inside any script:
```bash
creds = {
  "operatorUN": "yourldap",
  "accessChoice": "check/remove/grant",
  "MOCK_MODE": True
}
```

# 🧑‍💻 Production Mode (Post-Setup) 
Once credentials are configured on the SOC Cloud Workstation: 
```bash 
# From the workstation, run the CLI version with: 
python access_management.py 

# Or launch the GUI: 
python soctool.py 
```
The GUI wraps the CLI, handling input validation and passing the `--mock` flag automatically as needed.

# 📎 Sample CLI Usage 
```bash 
# Check a user's access 
python access_management.py check jdoe 

# Remove access with mock mode 
python access_management.py remove jdoe --mock 

# Grant access with RT ticket 
python access_management.py grant jdoe --rt_ticket 123456 
```
# 👀 Visibility & Audit 
- All actions are logged to Logs/access_management.log 
- RT ticket integration via rt_api.py supports adding comments or triggering escalations 
- Each script logs its own activity independently 

# 🔐 Security Notes 
- No secrets are ever stored in GitHub or in the codebase 
- MOCK_MODE is default unless explicitly overridden 
- Production credentials are stored only on the Cloud Workstation and Bitwarden 
- Scripts follow principle of least privilege when interacting with systems
